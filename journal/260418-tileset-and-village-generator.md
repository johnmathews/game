# 2026-04-18 — Tileset Setup and Village Generator

## What happened

Set up the Godot project with Kenney CC0 asset packs and created a procedural village generator for Spaarndam.

## Asset packs integrated

All from kenney.nl, all CC0 (public domain):

- **Isometric buildings** — 130 building tiles (houses, shops, roofs)
- **Isometric roads** — 97 tiles (roads, grass, intersections, dead-ends)
- **Isometric water** — 43 tiles (rivers, bridges, canals, waterfalls)
- **Isometric miniature library** — props and decorations
- **Pixel platformer** — 183 tiles at 18x18px for side-scrolling levels
- **Platformer characters** — 5 character types (Adventurer, Player, Soldier, Zombie) with animations

## Gridded spritesheets

The Kenney assets ship as individual PNGs and irregularly packed spritesheets. Godot's TileSetAtlasSource requires a regular grid, so we generated two properly gridded spritesheets:

- `ground_tileset.png` — 10x14 grid, 100x65px cells (roads, water, trees)
- `buildings_tileset.png` — 10x13 grid, 133x130px cells (buildings)

These are generated by a Python script using Pillow. The cells are padded/centered to fit the grid.

## Village generator

Created `village_generator.gd` — a `@tool` script that procedurally generates a Spaarndam village layout. Triggered via an `@export var generate` checkbox in the editor Inspector.

The generated layout includes:
- Spaarne river flowing north-south with a gentle curve
- 3 bridges crossing the river
- Main street parallel to the west bank
- Cross streets connecting to the river
- Church near the village center
- ~100 houses lining the streets on both banks
- Trees along the riverbank and outskirts
- Small park south of the church
- East bank neighborhood across the river

## Lessons learned

- Godot's TileSetAtlasSource needs every tile cell explicitly registered with `col:row/0 = 0` entries in the .tscn file, or the scene fails to load.
- The TileSetAtlasSource sub-resources must be declared before the TileSet that references them (Godot parses .tscn top-to-bottom).
- `@tool` scripts need the editor to re-parse them; sometimes a project reload is needed for `@export` vars to appear.
- Node paths in `@tool` scripts attached to child nodes need `../` to reach siblings.
